# aws.nvim — Developer Guide

This document covers the development workflow, project architecture, and
instructions for adding new views. For end-user installation, see
[README.md](README.md).

---

## Prerequisites

- Node.js >= 16
- npm
- Neovim >= 0.9 (with the Node.js remote plugin provider available)

---

## Development Setup

```bash
# 1. Clone the repo
git clone https://github.com/rogerterrazas/aws.nvim
cd aws.nvim

# 2. Install dependencies
npm install

# 3. Compile TypeScript
npm run build
```

Then, in Neovim:

```vim
:UpdateRemotePlugins
```

Restart Neovim. The `:NvimAws` command is now available.

## Development Workflow

### After making code changes

```bash
npm run build
```

If you added or removed a registered command, function, or autocmd, also run in
Neovim:

```vim
:UpdateRemotePlugins
```

Then restart Neovim.

> In practice, adding a new _view_ (registered via `viewRegistry.register()`)
> does not require `:UpdateRemotePlugins` because the `:NvimAws` command itself
> does not change. Only changes to top-level command/function/autocmd
> declarations require re-running `:UpdateRemotePlugins`.

### Watch mode (recompile on save)

```bash
npm run watch
```

You still need to reload the plugin in Neovim after each rebuild (restart or
`:source %` on the rplugin entry point).

### Running tests

```bash
npm test                # run once
npm run test:watch      # watch mode
npm run test:coverage   # with coverage report
```

### Linting and formatting

```bash
npm run check           # lint + format (auto-fix)
npm run check:ci        # lint + format (check only, no writes — used in CI)
```

## Architecture

### How the plugin loads

Neovim's rplugin system spawns a Node.js process and communicates over
msgpack-RPC. The entry file is `rplugin/node/nvim-aws.js`, which re-exports
`dist/index.js`. `src/index.ts` registers a single `NvimAws` command and all
eight views with the `ViewRegistry`.

### Dispatch chain

```
:NvimAws [args]
  → rplugin/node/nvim-aws.js         (static re-export shim)
  → dist/index.js                    (compiled src/index.ts)
  → router()                         (parses args: route | action | default)
  → handleRoute() / handleAction()
  → ViewRegistryEntry.initialize()   (creates buffer, populates content)
    or ViewRegistryEntry.actions[name]()
  → Neovim nvim_* API calls via plugin.nvim
```

### View registry pattern

Each view is a `ViewRegistryEntry` object:

```typescript
export interface ViewRegistryEntry {
  name: ViewName;
  bufferLabel: string | ((...args: string[]) => string);
  initialize: (
    plugin: NvimPlugin,
    window: Window,
    args: string[],
  ) => Promise<void>;
  actions: Record<string, ActionHandler>;
}
```

- `initialize` creates a `nofile` buffer, populates it, sets its filetype and
  name, and configures buffer-local keybindings.
- `actions` maps action name strings to handler functions.

### Buffer naming

All buffers are named `nvim-aws | <Label>` (no profile) or
`nvim-aws | <Label> | <profile> | <region>` (with profile). The `nvim-aws |`
prefix lets the router detect plugin-owned buffers.

### Buffer reuse and deduplication

`handleRoute()` checks whether a named buffer already exists and has content;
if so, it reuses it. A `Set<string>` called `inFlightRoutes` prevents duplicate
async `initialize()` calls for the same route+args combination.

### Session state

`src/session/index.ts` holds a single module-level `activeProfile: AwsProfile | null`.
All AWS SDK client factories read this to configure credentials and region.

## Adding a New View

### 1. Create the view files

```
src/views/<service>/<view>/
├── <view>.ts        # ViewRegistryEntry export (initialize + buffer setup)
└── commands.ts      # Action handler functions
```

Use an existing view (e.g., `src/views/ddb/tables/`) as a reference.

### 2. Export a `ViewRegistryEntry`

```typescript
// src/views/myservice/myview/myview.ts
import type { ViewRegistryEntry } from "../../../types/index.js";
import { myAction } from "./commands.js";

export const myViewEntry: ViewRegistryEntry = {
  name: "my_view_name",
  bufferLabel: "My View Label",
  initialize: async (plugin, window, args) => {
    // create buffer, set content, set filetype, register keybindings
  },
  actions: {
    myAction,
  },
};
```

### 3. Add the view name to the `ViewName` union type

```typescript
// src/types/index.ts
export type ViewName =
  | "aws_home"
  | "aws_accounts"
  | "dynamo_db_tables"
  // ... existing views ...
  | "my_view_name"; // add this
```

### 4. Add the filetype mapping

```typescript
// src/router/index.ts — FILETYPE_TO_VIEW map
'nvim-aws-my-view': 'my_view_name',
```

### 5. Register the view in `src/index.ts`

```typescript
import { myViewEntry } from "./views/myservice/myview/myview.js";

viewRegistry.register(myViewEntry);
```

### 6. Build and verify

```bash
npm run build
```

In Neovim:

```vim
:NvimAws route my_view_name
```

## Logging

Use the logger in any source file:

```typescript
import { logger } from "../../utils/logger.js";

logger.info("something happened");
logger.error("something went wrong", error);
```

Output is written to `~/.local/state/nvim/nvim-aws.log`.
